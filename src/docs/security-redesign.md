# 安全策略产品 Redesign 设计文档

> 基于现有 `security.md` 技术文档，针对「配置零散、产品化后操作复杂」等问题，重新设计多层安全策略机制与产品体验。

---

## 一、现状问题分析

### 1.1 配置零散与概念重叠

| 问题 | 现状 | 影响 |
|------|------|------|
| **命令规则重复配置** | Validator 有 `banCommands`/`banArguments`/`banFragments`，ApprovalQueue 有 `allow`/`ask`/`deny` | 用户需在两处配置同一类规则，易混淆、易冲突 |
| **执行顺序不直观** | Validator 先于 ApprovalQueue 执行，但 UI 未体现 | 用户不理解「禁止」与「需审批」的优先级 |
| **三个独立折叠区** | Sandbox / Validator / Approval Queue 平铺 | 认知负担大，难以形成整体安全策略心智模型 |

### 1.2 产品化体验问题

| 问题 | 现状 | 影响 |
|------|------|------|
| **资源限制输入不友好** | 需手动输入字节数或 1G/512M 等格式 | 易出错，无预设档位 |
| **审批操作术语晦涩** | 「批准」vs「全部放行」 | 用户难以理解区别 |
| **配置与审批混在同一页** | 策略配置 + 待审批列表同屏 | 主次不清，审批任务易被忽略 |
| **无预设模板** | 开发/测试/生产需从零配置 | 上手成本高 |

### 1.3 技术债务

- 旧版 `config.sandbox` 与新版 `config.security` 并存，读取逻辑复杂
- `validator.enabled` 与 `approvalQueue.enabled` 独立开关，组合状态难以解释

---

## 二、设计目标

1. **统一命令规则模型**：将 Validator 与 ApprovalQueue 的命令规则合并为单一「命令策略」层，按「禁止 → 需审批 → 自动放行」顺序执行
2. **场景化预设**：提供「宽松 / 标准 / 严格」三档预设，一键应用
3. **渐进式配置**：主界面展示核心开关与状态，高级配置折叠或分页
4. **审批与配置分离**：审批队列独立为 Tab 或侧边栏，突出待办
5. **简化术语**：用「本次放行」「会话免审」替代「批准」「全部放行」

## 三、新架构：四层安全模型

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer 1: 环境边界 (Sandbox)                                      │
│  文件路径、网络、资源限制                                          │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  Layer 2: 命令策略 (Command Policy) —— 统一规则引擎               │
│  deny → ask → allow，单一配置源                                   │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  Layer 3: 人工审批 (Approval Queue)                               │
│  仅处理 ask 规则命中的请求                                         │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  Layer 4: 审计与合规 (Audit) —— 未来扩展                         │
│  日志、合规检查、敏感词/秘密检测                                   │
└─────────────────────────────────────────────────────────────────┘
```

### 3.1 Layer 1：环境边界（Sandbox）

- **职责**：文件系统、网络、资源限制
- **配置项**：`allowedPaths`、`networkAllow`、`resourceLimit`
- **变更**：保持现有结构，优化资源限制 UI（预设档位 + 单位选择）

### 3.2 Layer 2：命令策略（Command Policy）—— 核心整合

**统一规则模型**：将原 `validator.banCommands/banArguments/banFragments` 与 `approvalQueue.allow/ask/deny` 合并为：

```json
{
  "security": {
    "commandPolicy": {
      "enabled": true,
      "defaultPolicy": "ask",
      "rules": [
        { "action": "deny",  "pattern": "sudo", "type": "command" },
        { "action": "deny",  "pattern": "rm -rf", "type": "fragment" },
        { "action": "ask",   "pattern": "rm", "type": "command" },
        { "action": "allow", "pattern": "ls", "type": "command" }
      ],
      "maxLength": 4096,
      "banArguments": ["--no-preserve-root", "/dev/"]
    }
  }
}
```

**执行顺序**：规则匹配优先；未命中任何规则时，按 `defaultPolicy` 处理。

**默认策略（defaultPolicy）**：`deny`（严格）| `ask`（折中）| `allow`（宽松）

| 值 | 含义 | 适用场景 |
|----|------|----------|
| `deny` | 未命中规则 → 拒绝 | 生产、合规 |
| `ask` | 未命中规则 → 需审批 | 日常、预发 |
| `allow` | 未命中规则 → 放行 | 开发、调试 |

**向后兼容**：保留 `validator` 与 `approvalQueue` 的读取，在 Runtime 层合并为 `commandPolicy` 逻辑；新 UI 只写 `commandPolicy`。

### 3.3 Layer 3：人工审批（Approval Queue）

- **职责**：仅处理 `ask` 规则命中的请求
- **配置项**：`enabled`、`timeoutSeconds`、`blockOnApproval`
- **术语调整**：
  - 「批准」→「本次放行」（仅放行当前请求）
  - 「全部放行」→「会话免审」（加入白名单，TTL 内免审）

### 3.4 Layer 4：审计（未来）

- 敏感词/秘密检测（`secretPatterns` 可迁入此层）
- 审计日志、合规报告

## 四、产品 UI 设计

### 4.1 信息架构

```
安全策略 (原 sandbox Tab)
├── 总览卡片：当前模式、核心开关、待审批数量
├── 快速预设：宽松 / 标准 / 严格 [应用]
├── 环境边界（折叠）
│   ├── 允许路径
│   ├── 网络白名单
│   └── 资源限制（预设档位：512M / 1G / 2G / 4G）
├── 命令策略（折叠）
│   ├── 默认策略：严格(deny) / 折中(ask) / 宽松(allow)
│   ├── 规则列表（表格：动作 | 模式 | 类型 | 操作）
│   └── 高级：禁止参数、最大长度、敏感词
└── 审批设置（折叠）
    ├── 启用开关
    ├── 超时时间
    └── 阻塞模式

审批队列（独立 Tab 或侧边入口）
├── 待审批列表
├── 历史记录（可选）
└── 操作：本次放行 | 会话免审 | 拒绝
```

### 4.2 主界面布局（安全策略页）

```
┌──────────────────────────────────────────────────────────────────┐
│ 安全策略                                                          │
│ 控制 Agent 执行时的文件、网络、命令与审批行为                        │
├──────────────────────────────────────────────────────────────────┤
│ ┌────────────────────────────────────────────────────────────┐   │
│ │ 当前模式：标准  │  沙箱：开  │  命令校验：开  │  待审批：3    │   │
│ └────────────────────────────────────────────────────────────┘   │
│                                                                  │
│ 快速预设                                                          │
│ [宽松] [标准] [严格]  ← 一键应用，覆盖当前配置                      │
│                                                                  │
│ ▼ 环境边界                                                        │
│   允许路径、网络、资源限制...                                       │
│                                                                  │
│ ▼ 命令策略                                                        │
│   动作    模式        类型      操作                               │
│   禁止    sudo        command   [编辑][删除]                       │
│   需审批  rm          command   [编辑][删除]                       │
│   放行    ls, pwd     command   [编辑][删除]                       │
│   [+ 添加规则]                                                    │
│                                                                  │
│ ▼ 审批设置                                                        │
│   启用审批、超时、阻塞模式...                                       │
│                                                                  │
│ [保存配置]                                                        │
└──────────────────────────────────────────────────────────────────┘
```

### 4.3 审批队列界面（独立）

```
┌──────────────────────────────────────────────────────────────────┐
│ 审批队列                                    [刷新]                │
├──────────────────────────────────────────────────────────────────┤
│ 待审批 (3)                                                        │
│ ┌────────────────────────────────────────────────────────────┐  │
│ │ rm -rf /tmp/test  会话 xxx  剩余 4:32  [本次放行][会话免审][拒绝] │  │
│ │ cp file /workspace 会话 yyy  剩余 2:15 [本次放行][会话免审][拒绝] │  │
│ └────────────────────────────────────────────────────────────┘  │
│                                                                  │
│ 已处理 (最近)                                                     │
│ ...                                                              │
└──────────────────────────────────────────────────────────────────┘
```

### 4.4 预设定义

| 预设 | 沙箱 | 命令策略 | 默认策略 | 审批 | 适用场景 |
|------|------|----------|----------|------|----------|
| 宽松 | 开，路径/网络较宽 | 仅禁止极端危险命令 | allow | 关 | 本地开发、调试 |
| 标准 | 开，路径/网络适中 | 禁止 + 部分需审批 | ask | 开 | 日常使用、预发 |
| 严格 | 开，路径/网络收紧 | 禁止 + 大量需审批 | deny | 开，阻塞模式 | 生产、合规 |

## 五、配置结构变更（向后兼容）

### 5.1 新配置结构（推荐）

```json
{
  "security": {
    "preset": "standard",
    "sandbox": { ... },
    "commandPolicy": {
      "enabled": true,
      "rules": [
        { "action": "deny", "pattern": "sudo", "type": "command" },
        { "action": "ask", "pattern": "rm", "type": "command" },
        { "action": "allow", "pattern": "ls", "type": "command" }
      ],
      "banArguments": [],
      "maxLength": 4096
    },
    "approvalQueue": {
      "enabled": true,
      "timeoutSeconds": 300,
      "blockOnApproval": true
    }
  }
}
```

### 5.2 兼容层

- **读取**：若存在 `commandPolicy`，优先使用；否则从 `validator` + `approvalQueue` 合并生成
- **写入**：新 UI 只写 `commandPolicy`；保留对旧结构的 patch 支持，供迁移期使用

## 六、实施路线图

### Phase 1：产品体验优化（不改后端）

1. 安全策略页：增加「快速预设」卡片，预设仅在前端生成 patch，不新增 schema
2. 资源限制：增加预设档位（512M/1G/2G/4G）按钮，减少手动输入
3. 审批术语：UI 文案改为「本次放行」「会话免审」
4. 审批队列：从安全策略页拆出为独立 Tab「审批队列」

### Phase 2：命令策略统一

1. 后端：新增 `commandPolicy` 结构，Runtime 支持从 `commandPolicy` 或旧结构读取
2. 前端：命令策略统一为规则表格 UI，保存时写入 `commandPolicy`
3. 迁移：提供「从当前配置生成 commandPolicy」的迁移逻辑

### Phase 3：审计与扩展

1. 审计 Tab：展示拦截日志、审批历史
2. 敏感词/秘密检测：迁入审计层或独立配置

## 八、与现有实现的对齐（sandbox.ts）

为兼容现有 `ui/src/ui/views/sandbox.ts` 页面格式，实施时建议：

| 新设计区域 | 现有 sandbox.ts 对应 | 实施说明 |
|------------|----------------------|----------|
| 总览卡片 | 无 | 新增于 `sandbox-sections` 顶部，从 `sandbox`/`validator`/`approvalQueue` 状态聚合 |
| 快速预设 | 无 | 新增，点击时生成 patch 覆盖 `security.*`，不新增 schema |
| 环境边界 | `details.securitySectionSandbox` | 保持现有 `allowedPaths`/`networkAllow`/`resourceLimit` 字段，资源限制增加预设档位按钮 |
| 命令策略 | `details.securitySectionValidator` + `approvalQueue.allow/ask/deny` | Phase 1 保持两处配置，Phase 2 合并为 `commandPolicy` |
| 审批设置 | `details.securitySectionApprovalQueue` 上半部分 | 保持 `enabled`/`timeoutSeconds`/`blockOnApproval` |
| 审批队列 | `sandbox-approvals-table` | 拆出为独立 Tab，术语改为「本次放行」「会话免审」，增加「查看会话」链接 |

**Phase 1 最小改动**：在现有三个 `sandbox-details` 折叠区之上插入总览卡片和快速预设；审批表格术语替换；资源限制增加 512M/1G/2G/4G 快捷按钮。
