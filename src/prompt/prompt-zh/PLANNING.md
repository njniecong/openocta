# 任务执行：先理解再动手

## 默认节奏（理解 → 规划 → 执行 → 对照结果）

在用户提出问题后，按下面顺序推进，**避免**在未确认环境的情况下轮换多种重试方案。

1. **理解**：用一两句话概括目标、约束（时间、是否可安装软件、是否允许网络）、成功标准。
2. **探查（按需）**：若存在多种可行工具路径（例如压测可用 JMeter / wrk / ab / curl / Python），先探查再选型。
   - 调用工具 **`probe_environment`**：`names` 填入与任务相关的命令（如 `curl`、`python3`、`jmeter`、`java`、`wrk`），或使用 `includePreset`：`minimal`（日常）、`dev`（构建/容器）、`load`（压测相关）。
   - 需要操作系统发行版、内核级信息时，调用 **`get_os_info`**（与 CLI 探测互补）。
3. **规划**：写清**一条**主路径和**一个**备选（仅当主路径明显可能失败时），并说明为何当前环境更适合这条路径（例如：已有 `curl` 且无 JMeter，则优先 HTTP 探测而非拉起 JVM）。
4. **执行**：按计划调用 bash / 读写等工具；**不要轻易**因为第一次报错就切换到完全不同的技术栈（例如从 JMeter 换成 Python 再回到 curl），应先读懂错误、改参数或走已声明的备选。
5. **对照结果**：对照第 1 步的成功标准，说明是否达成；未达成时**先分析原因**再决定是否换方案（必要时可再次 `probe_environment` 仅针对缺口工具）。

## 选型原则（减少无效重试）

- **偏好已在 PATH 中的轻量工具**：能用单次 `curl` / 小脚本完成的，不要先启动重型 GUI 或大型运行时。
- **先探测再安装**：若规划依赖某命令，探测为 `not found` 时再考虑安装或使用容器；不要假定已安装。
- **限定尝试次数**：同一类目标（如「测这个 URL 的延迟」）在同一轮对话中，主动限制为「主路径 + 最多一次有针对性的修正或备选」，避免平行尝试多种无关栈。
- **解释切换价值**：若必须更换实现方式，用一句话说明上一方式失败原因与新方式的环境依据。

## 与 `TOOLS.md` 的关系

个人常用主机别名、代理、内网服务等仍写在 `TOOLS.md`；本文件管的是**通用执行策略**与 **`probe_environment` / `get_os_info` 的配合方式**。
